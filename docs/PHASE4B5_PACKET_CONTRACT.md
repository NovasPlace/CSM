# Phase 4B.5: Experience Packet Contract Lock

## Three-Layer Separation

```
+---------------------------+--------------------------------------------+
| Layer                     | Purpose                                    |
+---------------------------+--------------------------------------------+
| experience_packets        | Observation / interpreted state only       |
| memory_candidate_queue    | Promotion / review only                    |
| memories                  | Confirmed durable knowledge only           |
+---------------------------+--------------------------------------------+
```

### Layer 1: `experience_packets` (observation)
- Created per-tool-event via the passive capture path (`tool-execute.ts` after-hook)
- Stores **derived internal state** as JSON: `cognitiveLoad`, `frustration`, `energy`, `dominantEmotion`, `stance`, `urgency`
- Stores **raw signals** as JSON: `toolName`, `argsPreview`, `exitCode`, `error`, `intent`, `callCount`
- Entry types: `tool_execution`, `error`, `milestone`, `decision`, `session_start`, `session_end`, `distill_group`, `loop_signal`
- **Never produces a candidate or memory** — observation layer only
- Readable via `csm_memory_packets` tool (audit, not promotion)

### Layer 2: `memory_candidate_queue` (promotion / review)
- Advisory candidates generated by `csm_memory_candidate_generate`
- Reads from `experience_packets` (and other tables) but does not write to them
- Each candidate has a `candidate_type`, `memory_id`, `reason`, `confidence`, and `status`
- Promotion pipeline (Phase 4C) reads packets, evaluates promotion-worthiness, writes candidates

### Layer 3: `memories` (confirmed durable knowledge)
- Only written by explicit save, approval, or promotion decision
- **Never written directly by the packet layer**
- Promotion must go through `memory_candidate_queue`

## Stable Deriver Vocabulary

The `internal_state` JSON field across both PG and SQLite uses these field names:

| Field             | Type      | Range    | Description                                      |
|-------------------|-----------|----------|--------------------------------------------------|
| cognitiveLoad     | number    | 0.0-1.0  | Perceived cognitive effort (recent calls, errors) |
| frustration       | number    | 0.0-1.0  | Error/recovery pressure signal                   |
| energy            | number    | 0.0-1.0  | Momentum (decays on errors, boosts on milestones) |
| dominantEmotion   | string    | enum     | neutral / frustration / success / curiosity / concern |
| stance            | string    | enum     | exploratory / focused / recovery / stuck         |
| urgency           | number    | 0.0-1.0  | How critical/speed-required the situation feels  |

These are pure deterministic functions in `internal-state-deriver.ts`. No LLM calls. No DB reads.

## Promotion-Worthiness Signals (for Phase 4C)

Clients evaluating promotion should consider:

1. **High frustration + recovery state** → may indicate lesson-worthy event
2. **High urgency + low energy** → may indicate need for intervention
3. **Sustained exploratory stance** → broad learning, not specific promotion
4. **Milestone/success events** → low promotion priority (already resolved)
5. **Loop signals** → high promotion priority (frustration pattern, lesson-worthy)
6. **Decision events** → medium priority (may encode procedure/episodic knowledge)

## Schema Vocabulary Mapping (PG ↔ SQLite)

| PG type         | SQLite type         | Notes                          |
|-----------------|---------------------|--------------------------------|
| BIGSERIAL       | INTEGER AUTOINCREMENT | PK generation                 |
| TEXT            | TEXT                | Identical                      |
| JSONB           | TEXT                | Stored as JSON string in SQLite|
| REAL            | REAL                | Identical                      |
| TIMESTAMPTZ     | TEXT                | ISO 8601 string in SQLite      |

All column names, CHECK constraints, and indexes are identical between PG and SQLite schemas.
